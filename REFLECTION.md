## 一、Superpowers 技能评价

### 1.1 最有价值的技能：Brainstorming 与 Writing-Plans

在本项目中，**Brainstorming** 和 **Writing-Plans** 是最核心的两个技能，它们直接决定了项目质量的上限。

Brainstorming 的价值在于它**强化了人与机器的交互深度**。在项目启动阶段，我面对的是一个模糊的想法——"做一个编码 agent 的 harness"。但通过 Brainstorming 的多轮追问，Agent 帮我梳理出了 11 状态 FSM 的架构、反馈闭环作为重点维度的选择、护栏系统的分层设计等关键决策。在这个过程中，Agent 提出了 7 个重要修正（如 HITL 独立状态化、TOOL_ERROR 状态分离、Classifier 规则驱动等），这些修正如果我独自设计很可能会遗漏。

更重要的是，Brainstorming 的**多次交互特性**解决了一个实际问题：人在项目初期倾向于"少量多次"的信息反馈，而不是一次性给出完整规约。每次 Agent 的追问都在帮我加深对项目的理解，迫使我思考"这个决策的 trade-off 是什么"、"为什么选方案 B 而不是方案 A"。这种逐步深化的对话，比一个人闷头写文档要高效得多。

Writing-Plans 的价值在于它为后续的 Subagent 工作提供了**必要的组织框架**。没有一份颗粒度合理的 PLAN，Subagent 就像没有地图的探险者。12 个 Task 的划分、依赖关系标注、验证步骤的明确，使得后续的 TDD 实现有了清晰的执行路径。

### 1.2 形式大于实质的技能

其他的技能（如 TDD、Subagent-Driven Development、Code Review 等）在一般的 Agent 工作流中也能较好地执行，它们在 Superpowers 框架中被体系化了，但并非不可替代。这并不是说它们没有价值，而是说它们更接近"工程纪律的自动化执行"，而不是 Brainstorming 那种"人机协作的创造性过程"。

## 二、TDD 在 AI 协作下的体验

TDD（测试驱动开发）在本项目中是一个**放大器而非阻碍**。原因在于项目可以自然地分为两个阶段：

1. **核心编码阶段**（Task 1-12）：实现 Harness 内核——状态机、工具系统、护栏、反馈闭环等
2. **表现层优化阶段**（Task 13-18）：添加 CLI 交互、Renderer 抽象、可观测性等

TDD 的"先写失败测试 → 再写实现 → 再重构"流程，在第一阶段起到了**保护核心层代码**的作用。每次 Subagent 实现一个新模块时，先写测试再写代码的流程确保了：
- 模块的接口在实现之前就被明确了
- 回归测试可以捕获后续修改引入的破坏
- 测试本身就是文档，说明了模块的预期行为

特别是在第二阶段的表现层开发中，TDD 确保了核心层的 53 个测试一直保持通过，没有被误改破坏。

## 三、SPEC/PLAN 对实现质量的影响

SPEC 和 PLAN 在本项目中起到了**定位任务方向**的锚定作用。每个 Subagent 被派发时，其任务描述来源于 PLAN 中的 Task 定义，而 Task 的接口和验证步骤又来源于 SPEC 中的设计。

在多层架构中（父 Agent → Subagent），即使 Subagent 的理解出现偏差，上层还有父 Agent 的 Review 环节兜底。因此，**偏离问题的实际影响感知并不大**——Review 环节会捕获方向偏差，并指导 Subagent 修正。

这一点与我的预期不同：我原本以为 SPEC/PLAN 的模糊会频繁导致 Subagent 做错事，但实际上 Review 机制消化了大部分偏差。当然，这也说明 SPEC/PLAN 的质量仍然重要——它们决定了偏差发生的频率和 Review 的工作量。

## 四、凭据与分发：被忽视的工程要求

课程要求中的凭据安全存储和分发设计，最初我认为是"边角料"工作，但实际上它们**迫使我想清楚了很多原本会忽略的问题**：

- **架构问题 vs 逻辑问题的区分**：在接入真实 API 进行端到端验证时，暴露了 3 个 Bug。其中 Bug 3（Context 从未填充）是架构设计缺陷，而 Bug 2（状态机转移非法）是逻辑错误。这种区分不是凭据和分发本身带来的，而是在配置凭据、构建分发产物的过程中，必须把整个系统跑通一遍，自然会暴露深层问题。

- **改动的连锁反应**：凭据存储方案（`.env` + `getpass` 引导）看起来简单，但它要求：代码中不能有硬编码的 key、`.env` 必须在 `.gitignore` 中、首次运行要有引导流程。这些要求迫使我在设计初期就考虑"这个改动是否简单？是否会为后续埋雷？"——这种思维方式在凭据安全的场景下被强化了，但在其他地方同样适用。

## 五、AI4SE 方法论的批判性思考

### 5.1 价值所在

AI4SE 这门课的核心价值在于回答两个问题：

1. **如何在 AI 辅助下更好地了解自己的项目？** Brainstorming 的追问机制迫使你思考那些你原本不会想到的问题（"HITL 应该是一个状态还是 Guardrail 的分支？"、"Feedback Loop 的边界在哪里？"）。这个过程本身就是一种"被迫深入理解"。

2. **如何让 AI 做出更符合自己实际需求的产出？** 这不是靠更好的提示词工程，而是靠 SPEC + PLAN + Review 的工程化流程。AI 的产出质量取决于你给它提供的约束质量。

### 5.2 核心担忧：规约是否会成为 AI 的脚镣？

随着模型能力的疯狂提升，一个值得深思的问题是：**当前这种"用详尽规约约束 AI"的方式，是否会在未来成为 AI 的脚镣？**

假设三年后，AI 的编码能力比现在提升 10 倍，一个模糊的描述就能生成高质量代码——那么我们现在投入大量精力编写的 SPEC 和 PLAN，会不会反而限制了 AI 的发挥空间？就像给一个奥运短跑选手画了一条 10 米的跑道——他本可以跑得更远。

当然，现在的现实是：AI 仍然需要明确的约束来保证产出质量。但这个矛盾会随着模型能力的提升而加剧。

### 5.3 人的上限依然重要

无论 AI 如何发展，**人的上限（human ceiling）仍然是项目质量的最终决定因素**。具体体现在：
- 辨别 AI 输出质量的能力
- 发现架构层面缺陷的洞察力
- 在关键决策点上做出正确选择的能力

在未来，**人与人的沟通**（需求传递、团队协作）和**人与 AI 的沟通**（约束设计、结果审查）会成为核心能力。Superpowers 框架在这方面提供了有价值的训练——它不是在教你怎么用 AI，而是在教你怎么在 AI 时代做工程。

## 六、如果重做，我会改变什么

如果重新开始这个项目，我会在以下方面做出调整：

1. **更早进行真实 API 验证**：Mock 模式下的测试通过后，应该更早地用真实 API 跑一遍端到端流程。3 个 Bug 暴露得越晚，修复成本越高。

2. **减少 Serial Review 的等待**：当前 12 个 Task 按顺序串行执行，虽然有简化 Review 的好处，但效率较低。如果 Task 之间没有依赖关系，应该真正并行化。

3. **更关注 Context Engineering 层**：Bug 3 暴露的 Context Engineering 缺失是项目中最有价值的发现。如果在设计阶段就意识到"状态数据和模型输入之间的桥梁"是独立职责，早期的架构会更干净。

## 七、总结

本项目通过 Superpowers 框架完成了一个完整的 Coding Agent Harness 开发周期。最大的收获不是代码本身，而是对"AI 时代工程方法论"的批判性理解：

- Brainstorming 和 Writing-Plans 是当前最有价值的技能，因为它们解决了"人如何与 AI 高效协作"的核心问题
- TDD 在 AI 协作下是有效的保护机制，特别是在分层架构中保护核心层不被误改
- 凭据和分发这类"非功能性需求"在实践中迫使开发者思考更深层的工程问题
- Context Engineering 层是"状态数据 → 模型输入"的关键桥梁，早设计比晚修复成本低得多
- 随着 AI 能力的提升，"规约是否成为脚镣"的矛盾会越来越突出，这需要工程方法论的持续演进

最终，这个项目的核心启示是：**AI 不会降低对人的要求，它会改变对人的要求**——从"写代码"的能力，转向"设计约束、判断质量、做出决策"的能力。
